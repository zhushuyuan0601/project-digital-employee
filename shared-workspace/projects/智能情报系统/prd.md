# AI 情报自动收集系统 PRD

## 需求背景

**来源**：青峰（CEO）

**背景**：
- 目前每日情报简报依赖情报研究员手动爬取和整理，耗时约 1-2 小时
- AI 领域发展迅速，信息量大，人工筛选效率低
- 需要一个自动化系统来持续跟踪 AI 热点，提高情报收集效率

## 目标用户

### 用户画像
- **主要用户**：需要关注 AI 领域热点的创业者、产品经理、投资人
- **使用场景**：每天早上定时查看 AI 行业动态
- **核心痛点**：
  - 手动收集信息耗时耗力
  - 信息来源分散（公众号、新闻网站、技术博客）
  - 信息过载，难以筛选有价值的内容
  - 缺乏结构化的情报归档

### 预期收益
- 节省 70% 以上的人工整理时间
- 情报覆盖率提升（支持多个信息源）
- 情报质量提升（AI 自动分类和提炼）
- 可追溯的历史情报库

## 市场分析

### 市场规模
- AI 行业关注度持续上升，大量从业者和投资者需要行业情报
- 企业情报服务市场存在但价格高昂（如 Bloomberg、Gartner）
- 个人用户缺乏易用的自动化情报工具

### 竞品分析

| 产品 | 优势 | 劣势 |
|------|------|------|
| 人工情报服务 | 质量高、有深度 | 价格昂贵、更新慢 |
| RSS 订阅工具 | 免费、自动更新 | 需要手动筛选、无智能分类 |
| 新闻聚合平台 | 信息量大、覆盖广 | 噪音多、缺乏深度分析 |

### 差异化优势
- **专注 AI 领域**：垂直深度，避免噪音
- **智能分类**：自动识别大模型、智能体、产品等主题
- **自动化流程**：爬取→分类→生成简报→发送，全程自动化
- **本地化**：支持中文信息源（微信公众号、科技媒体）

#### 数据提取字段
```python
{
    "title": "文章标题（必填）",
    "url": "文章链接（必填，作为唯一键）",
```

- 请求间隔：随机 2-5 秒

## 5. 数据库设计

### articles 表
```sql
CREATE TABLE articles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    url TEXT UNIQUE NOT NULL,
    source TEXT,           -- 来源：36氪、量子位、智东西
    source_id TEXT,        -- 源站点标识
    content TEXT,          -- 内容摘要（前 500 字）
    full_content TEXT,     -- 完整内容
    category TEXT,         -- 分类：大模型、智能体、产品、竞品、学术
    category_confidence REAL DEFAULT 0.0,  -- 分类置信度（0-1）
    published_at TIMESTAMP,
    crawled_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    importance INTEGER DEFAULT 0,  -- 重要性评分（0-10）
    status TEXT DEFAULT 'new',     -- new/processed/sent
    error_message TEXT            -- 错误信息（如爬取失败）
);

CREATE INDEX idx_articles_category ON articles(category);
CREATE INDEX idx_articles_source ON articles(source);
CREATE INDEX idx_articles_published ON articles(published_at DESC);
CREATE INDEX idx_articles_status ON articles(status);
```

### categories 表
```sql
CREATE TABLE categories (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL,
    description TEXT,
    keywords TEXT  -- JSON 数组，关键词列表
);
```

###全文搜索索引
```sql
CREATE VIRTUAL TABLE articles_fts USING fts5(
    title, content, source,
    content='articles',
    content_rowid='id'
);
```

## 7. 技术方案（已确认）

### 技术栈

| 层级 | 技术 | 理由 |
|------|------|------|
| **编程语言** | Python 3.11+ | 生态丰富、上手快 |
| **爬虫引擎** | Playwright | 动态渲染、支持 JS |
| **数据存储** | SQLite + FTS5 | 轻量、支持全文搜索 |
| **AI 分类** | 本地 GLM-4/Qwen | 免费、准确、低延迟 |
| **定时调度** | APScheduler | 功能完善、易维护 |
| **邮件发送** | smtplib + SMTP | 免费、原生支持 |
| **飞书集成** | 飞书 Python SDK | 官方 SDK、稳定 |
| **日志** | Python logging | 原生、易扩展 |

### 关键技术决策

#### 爬取方案
- **MVP 阶段**：新闻网站使用 Playwright 爬取
- **微信公众号**：暂时跳过，Phase 2 接入第三方 API

#### 数据存储
- 使用 SQLite（轻量、单文件）
- 支持全文搜索（FTS5）
- 定期备份（每日 gzip 压缩）

#### AI 分类
- 使用本地 GLM-4 模型（local-qwen3/GLM-4.7-FP8）
- Few-shot Prompting 提高准确率
- 关键词匹配作为兜底

#### 定时任务
- APScheduler 管理内部任务
- 任务超时控制
- 失败重试机制（最多 2 次）

## 8. 风险与缓解

| 风险 | 等级 | 缓解方案 |
|------|------|----------|
| **爬虫反爬** | 高 | Playwright + 随机延迟 + User-Agent 轮换 |
| **AI分类准确率不足** | 中 | Few-shot Prompting + 关键词兜底 + 人工标注 |
| **数据源失效** | 中 | 多源备份 + 定期检查源可用性 |
| **定时任务执行失败** | 低 | 任务重试 + 备用调度（系统 cron） |
| **推送服务不可用** | 低 | 双渠道推送（邮件 + 飞书） |
| **系统崩溃** | 中 | 日志记录 + 健康检查 |
